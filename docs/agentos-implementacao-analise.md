# Análise da Implementação Atual vs Definições Refinadas ## Visão Geral Esta análise compara a implementação atual do PicoClaw com as definições refinadas propostas para as Etapas 6-9, identificando gaps e propondo ajustes necessários. ## Análise por Componente ### 1. Sistema de Orquestração Central (Etapa 6) #### Implementação Atual - **Bootstrapper** (`pkg/agentos/bootstrap.go`): Implementa bootstrap básico com SystemInstance - **SystemInstance**: Estrutura simplificada com componentes essenciais - **Fluxo**: Carrega manifesto → Valida → Conecta DB → Executa migrações → Provisiona atores → Cria engines #### Definição Refinada - **AgentOSEngine**: Engine unificada com gestão de ciclo de vida completo - **SystemInstance**: Instância rica com todos os subsistemas e métodos unificados - **ManifestStore**: Versionamento e gestão de manifestos - **Fluxo**: Detecta evolução → Planeja migração → Executa com segurança → Inicializa tudo #### Gaps Identificados 1. **Falta AgentOSEngine principal**: O Bootstrapper atual não tem a camada de abstração da AgentOSEngine 2. **SystemInstance incompleto**: Falta integração com EvolutionManager e métodos unificados 3. **Sem ManifestStore**: Não há versionamento de manifestos 4. **Sem detecção automática de evolução**: Não detecta mudanças entre versões #### Ajustes Necessários ```go // Adicionar AgentOSEngine principal type AgentOSEngine struct { config *EngineConfig instance *SystemInstance manifestStore *ManifestStore evolutionManager *EvolutionManager } // Estender SystemInstance type SystemInstance struct { // ... campos existentes ... EvolutionManager *EvolutionManager // Adicionar métodos unificados ExecuteOperation(ctx context.Context, opName string, params map[string]interface{}, actorID string) (map[string]interface{}, error) } // Adicionar ManifestStore type ManifestStore interface { SaveVersion(version *ManifestVersion) error GetVersion(version string) (*ManifestVersion, error) GetLatestVersion() (*ManifestVersion, error) ListVersions() ([]*ManifestVersion, error) } ``` ### 2. MCP Server com Paridade Total (Etapa 7) #### Implementação Atual - **MCPServer** (`pkg/mcp/server/server.go`): Implementa protocolo MCP básico - **OperationHandler**: Executa operações com validação de permissões - **Conversão**: Tools e resources gerados a partir do catálogo #### Definição Refinada - **Paridade Total**: Mesmas operações via API e MCP com mesma lógica - **Operation Bridge**: Ponte unificada entre API e MCP - **Resource Manager**: Gerenciamento avançado de recursos MCP #### Gaps Identificados 1. **SystemInstance não integrado ao MCP**: O MCPServer não usa SystemInstance para execução unificada 2. **Falta Operation Bridge**: Não há ponte formal entre API e MCP 3. **Métodos de execução separados**: API e MCP têm lógicas de execução diferentes #### Ajustes Necessários ```go // Modificar MCPServer para usar SystemInstance type MCPServer struct { instance *agentos.SystemInstance // Usar SystemInstance ao invés de componentes separados } // Adicionar Operation Bridge type OperationBridge struct { instance *agentos.SystemInstance } func (b *OperationBridge) ExecuteViaMCP(ctx context.Context, toolName string, args map[string]interface{}) (map[string]interface{}, error) { // Usar mesma lógica que API return b.instance.ExecuteOperation(ctx, toolName, args, b.getActorID(ctx)) } ``` ### 3. Gestão de Estados Complexos (Etapa 8) #### Implementação Atual - **WorkflowEngine** (`pkg/agentos/stateful/engine.go`): Implementa FSMs com persistência - **WorkflowInstance**: Gerencia estados com histórico - **GuardEvaluator**: Valida transições - **SideEffectExecutor**: Executa efeitos colaterais #### Definição Refinada - **StatefulWorkflowEngine**: Engine avançada com todos os recursos - **Sistema de Guardas Completo**: Com expressões e validações complexas - **Side Effects Avançados**: Com scripts, webhooks, notificações - **Timeouts e Histórico**: Gestão completa de tempo e auditoria #### Gaps Identificados 1. **WorkflowEngine básico**: Falta integração com SystemInstance 2. **Guard system limitado**: Sem suporte a expressões complexas 3. **Side effects simples**: Sem scripts customizados ou webhooks 4. **Sem timeout manager**: Não há gestão de timeouts automáticos #### Ajustes Necessários ```go // Estender WorkflowEngine type StatefulWorkflowEngine struct { instance *agentos.SystemInstance // Integrar com SystemInstance store *WorkflowStore guardEvaluator *GuardEvaluator effectExecutor *SideEffectExecutor timeoutManager *TimeoutManager historyManager *HistoryManager } // Estender GuardEvaluator type GuardEvaluator struct { // Adicionar suporte a expressões complexas } // Adicionar TimeoutManager type TimeoutManager struct { instance *agentos.SystemInstance } ``` ### 4.
